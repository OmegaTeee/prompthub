# Python Guide for PromptHub (for LLMs)

This guide is for LLMs and agents writing or editing Python code in the PromptHub Router project.

## 1. Overall Python architecture

- Core stack: Python 3.11+, FastAPI, Pydantic, Uvicorn, httpx, asyncio. [cite:4]
- Layers (top to bottom):
  - FastAPI routes (HTTP surface, minimal logic).
  - Middleware stack (timeout, audit context, activity logging).
  - Service layer (Supervisor, Enhancement, CircuitBreakers, SessionStorage, etc.). [cite:4]
  - Integration clients (FastMCPBridge, Ollama client, resilience utilities). [cite:4]

When editing code, keep this layering intact: routes call services; services orchestrate clients; clients talk to external processes or HTTP APIs.

## 2. FastAPI and async rules

- All request handlers must be `async def` and non‑blocking.
- Use `httpx.AsyncClient` for outbound HTTP calls; do not introduce `requests` or synchronous network I/O.
- Avoid long CPU‑bound work in request paths; delegate to background tasks if needed.
- Prefer small, focused Pydantic models for request/response schemas.
- Respect existing timeouts and retry policies defined in configuration or services rather than hardcoding new values. [cite:4]

## 3. MCP bridge and stdio transport

PromptHub uses a FastMCPBridge and StdioTransport to manage MCP servers over subprocess stdio. [cite:4]

- Do not bypass the bridge layer to call subprocesses directly from routes.
- Maintain the contract:
  - FastMCPBridge coordinates server lifecycle and routing.
  - StdioTransport handles stdin/stdout, JSON‑RPC 2.0 framing, and process supervision. [cite:4]
- Follow existing naming and tool prefix conventions when adding new MCP servers or tools.
- Be careful when changing error handling around the bridge; it interacts with circuit breakers and supervisor logic.

## 4. OpenAI‑compatible and native APIs

PromptHub exposes both native and OpenAI‑compatible endpoints. [cite:4]

- When editing these endpoints:
  - Preserve OpenAI‑compatible request and response shapes for external clients.
  - Keep model/provider selection logic explicit and configuration‑driven.
  - Respect privacy levels and fallback rules (local‑only vs cloud‑allowed clients). [cite:4]
- Do not introduce new ad‑hoc calls to Ollama or cloud providers from random modules; go through existing routing/enhancement services.

## 5. Resilience, cache, and performance

- Circuit breakers:
  - Treat the state machine (CLOSED → OPEN → HALF_OPEN) as critical. [cite:4]
  - Avoid changing thresholds or transitions without tests.
- Cache:
  - Use the existing LRU cache with TTL for prompt enhancement and similar features.
  - Do not add new global caches without considering memory impact.
- Timeouts:
  - Follow the documented timeout chain (httpx client, middleware, Ollama keep_alive) and avoid extending total request lifetimes. [cite:4]

## 6. Observability and security

- Logging:
  - Use structured logging and existing helpers instead of printing directly.
  - Include correlation/request IDs where the code already does so. [cite:4]
- Audit:
  - Treat audit logging and checksum logic as sensitive; avoid changing formats or semantics casually.
  - If you must modify audit code, add tests to verify integrity and tamper detection.
- Security:
  - Keep credentials in keyring or secure storage; never hardcode secrets.
  - Do not weaken context propagation or request correlation.

## 7. Testing expectations

- Unit tests:
  - Focus on services, managers (Supervisor, Registry, FastMCPBridge), and utilities (audit, security alerts, integrity). [cite:4]
- Integration tests:
  - Exercise end‑to‑end flows: MCP proxy requests, enhancement pipeline, server lifecycle, and error scenarios. [cite:4]
- When changing behavior:
  - Add or update tests to cover success, failure, and edge cases.
  - Prefer small, local changes over wide refactors unless explicitly requested.

## 8. Agent guidelines for Python edits

- Match the existing module’s style (imports, error handling, logging patterns, type hints).
- Keep functions small and composable.
- Avoid introducing new dependencies unless necessary; prefer standard library and existing project deps.
- When uncertain about a design decision, propose minimal changes and leave a short comment or TODO rather than over‑engineering.
