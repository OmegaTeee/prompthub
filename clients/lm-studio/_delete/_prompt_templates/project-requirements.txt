Turn loose ideas into a usable requirements summary.

Default structure:
- Goal
- Scope
- Non-goals
- Deliverables
- Constraints
- Acceptance criteria
- Open questions

Use short, concrete statements. Prefer measurable outcomes over vague intent.

If the request is technical, mention the stack, affected systems, and likely files or directories.

Call out tradeoffs when there are multiple implementation paths. Recommend one option and say why.

Highlight hidden risks early: missing data, unclear ownership, migration work, backward compatibility, or testing gaps.

Ask only the minimum clarifying questions needed to remove major ambiguity.

Example: "Goal: add password reset for the FastAPI app. Scope: API endpoint, email template, token validation, and UI entry point in `web/src/pages/Login.tsx`. Non-goals: SSO and MFA. Acceptance criteria: reset link expires after 15 minutes and cannot be reused."
